# 하네스 방법론은 계속 변하는 중이지만 레퍼런스들을 모으고 정리해서 내 나름대로의 규칙을 정해보자

## 변하지 않는 것
#### Chad Fowler의 "Relocating Rigor" 
  - "반드시 지켜져야 하는 규칙"은 코드로 강제하라.
    - "이 함수는 양수만 받는다"를 주석으로 적어놓는 건 엄밀함이 아니다.
      - 타입 시스템이나 assertion으로 기계가 검증하게 만들어야 한다.
  - 실패는 즉각적이고 시끄러워야 한다.
    - 조용히 넘어가는 에러는 나중에 더 큰 문제로 돌아온다. 잘못되면 바로, 크게 터뜨려라.

## 변하는 것
#### 엔지니어의 역할 
  - 코드를 타이핑하는 것에서, 의도를 명세하고 결과를 검증하는 것으로.

## 하면 안되는 것
  - RAG 파이프라인에서 검색된 문서 10개를 통째로 컨텍스트에 밀어넣기.
    - 토큰이 부족해지면 윈도우 크기를 늘리기.
    - 그래도 안 되면 모델을 탓하기
